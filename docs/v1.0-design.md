# Windy v1.0 — design memo

> Status: **shipped** in v1.0. v1.0 composes **F (wind speed)** and
> **D (IP collision — merge)**. This document captures the options
> considered and the rationale; the normative spec lives in `SPEC.md`
> §3.7 / §3.8.

## What v1.0 has to do

Through the v0.x line, Windy is — semantically — Befunge-98 with a
nicer surface and a few stricter promises. That's a fine *dialect*, but
"a new language" is a stronger claim than the spec currently earns.

For the major version to turn over, **Windy v1.0 must adopt at least
one semantic feature with no precedent in the Befunge family.** This
isn't about adding *more* — it's about adding *a thing the family
provably doesn't have*, so that "Windy is just Befunge with a haircut"
stops being a fair sentence.

Constraints any candidate has to clear:

1. **It has to fit the wind metaphor.** Windy is named after a wind
   mechanic; an addition that's tonally unrelated (e.g. an FFI shim) is
   the wrong move at this version cut.
2. **It can't paint over the existing 33 opcodes.** Programs written
   today should still run, with at most a documented compatibility
   note.
3. **It has to be tractably specifiable.** v0.x's whole identity is
   "the spec is the source of truth"; a v1.0 feature that requires a
   month of edge-case work before the spec is solid is a bad fit.
4. **It has to be *interesting to write programs against*.** The
   feature has to enable program patterns that didn't exist before, not
   just sit in §10 as a checkbox.

## Candidates

Five concrete directions. Each one is rated below on **fit** (does it
match the wind theme), **novelty** (does the Befunge family lack it),
**spec cost** (how much new SPEC text), **break risk** (do v0.x
programs still run), and **program-design payoff** (does it open new
patterns).

---

### A. Wind tension / inertia

**Idea.** The IP can no longer flip its direction freely. Direction
changes are constrained to the **graph of adjacent winds** — from any
of the eight winds you can rotate ±45°, but reversing 180° takes four
sequential turns. Existing direction-set opcodes (`→`, `↗`, `←`, etc.)
*request* a target direction; if that target is more than 45° from the
current heading the IP rotates one notch toward it and re-tries on the
next visit (or a new opcode `,` at a "drift" cell pre-rotates over a
gradient of cells).

**Sketch.** Add a single language-level invariant in §3:

> The IP's direction `(dx, dy)` is always one of the eight winds. A
> direction-set opcode that targets a wind more than one rotation away
> rotates the IP exactly one wind toward the target instead.

No new glyphs needed; the existing arrow set just gets re-interpreted.
Opt out via a `--free-winds` flag or a per-program directive for
backward compat.

| | |
|-|-|
| Fit | ★★★★★ — *defines* what "Windy" means |
| Novelty | ★★★★ — Befunge has nothing similar |
| Spec cost | ★★ — single rule, easy to write |
| Break risk | ★★★ — v0.x programs that flip 180° break unless `--free-winds` is on |
| Payoff | ★★★★ — routing becomes a real puzzle, programs look like wind currents |

### B. Time-aware grid

**Idea.** Cells are keyed `(x, y, t)`. The default `g` / `p` operate
at the current tick `t`. Two new opcodes, `G` and `P`, take an
explicit time offset:

```
y t G  →  push G[(x, y, t_now + t)]
v y t P →  G[(x, y, t_now + t)] := v
```

You can write a value into a *future* tick that no IP has reached yet,
or read a value that an IP wrote *in the past*. The grid stops being a
2D map and becomes a 3D recording.

**Sketch.** Two new opcodes, plus a §3.x on the time axis. The
already-sparse `HashMap<(x, y), BigInt>` becomes
`HashMap<(x, y, u64), BigInt>`; reads at `t_now` fall back to the
"all-time-default" plane (`HashMap<(x, y), BigInt>`) so initial
parsing of the source still produces values readable at every tick.

| | |
|-|-|
| Fit | ★★ — wind doesn't directly evoke time, but "weather forecasts" works |
| Novelty | ★★★★★ — esolangs with time-axis grids are rare |
| Spec cost | ★★★★ — needs a careful causal-consistency rule, paradox handling |
| Break risk | ★★ — purely additive opcodes |
| Payoff | ★★★★★ — opens *causal puzzles* foreign to Befunge |

---

### C. 2D stack

**Idea.** The stack is no longer a flat LIFO. It's a visible auxiliary
plane that grows in the direction of the IP at the moment of `push`:
heading east when you `push 5` puts the `5` one cell east of the
current "top" marker; heading north stacks it upward. `pop` reverses
the IP's last push direction.

**Sketch.** Replace §3.3 with a per-IP `stack_origin: (i64, i64)` and
`stack_top: (i64, i64)`. Push: `stack_top += dir; G[stack_top] = v`.
Pop: `v = G[stack_top]; G[stack_top] = SPACE; stack_top -= last_dir`.
The stack visually lives in a reserved region (e.g. `y < -1000`) and
the debugger renders it next to the program.

| | |
|-|-|
| Fit | ★★★ — wind currents pushing values around the plane is on-theme |
| Novelty | ★★★★ — Befunge has stack-of-stacks but not a *spatial* stack |
| Spec cost | ★★★★★ — every stack opcode (`+`, `:`, `\`, `$`, ...) needs new wording, multi-IP makes it hairy |
| Break risk | ★★★★★ — fundamentally changes data model |
| Payoff | ★★★★ — visual feedback in the debugger is gorgeous |

---

### D. IP collision semantics

**Idea.** When two live IPs land on the same cell on the same tick,
they don't pass through each other independently — they **interact**
per a defined rule. Likely options:

- **Annihilate**: both IPs die.
- **Merge**: stacks concatenate; direction becomes the (clipped) vector
  sum of the two; if the sum is zero, the merged IP dies.
- **Bounce**: each IP's direction reverses.

**Sketch.** Extends §3.5 with a "collision step" run after every
tick's IP pass. Probably needs a small tag on each IP so we can
distinguish "I started this tick at this cell" vs "I'm passing through
because two IPs collided here last tick".

| | |
|-|-|
| Fit | ★★★ — winds meeting can produce calm or storm; thematic |
| Novelty | ★★★★ — Befunge-98 has concurrent IPs but they ignore each other |
| Spec cost | ★★★ — well-bounded but needs careful tick semantics |
| Break risk | ★★ — only matters when `t` is used; v0.x split programs without collisions still work |
| Payoff | ★★★ — strategically interesting for multi-IP programs but not every program uses `t` |

---

### E. Cells as multi-token regions

**Idea.** A cell is no longer a single codepoint — it's a finite ordered
tuple of codepoints. The IP "consumes" one token per tick before
advancing. String mode (`"`) operates *inside* the cell tuple, not
across cells. Source files specify multi-token cells with a syntax like
`{abc}` to mean "a cell holding the tuple `('a', 'b', 'c')`".

**Sketch.** Big change to §3 (cell model) and §5 (file format). All
existing single-codepoint cells become 1-tuples implicitly; new syntax
opts into longer tuples.

| | |
|-|-|
| Fit | ★★ — doesn't really evoke wind |
| Novelty | ★★★ — bytecode-style packing in a 2D grid is unusual |
| Spec cost | ★★★★★ — touches parser, IR, debugger, file format |
| Break risk | ★★★ — syntax change; old files still parse |
| Payoff | ★★ — denser code, but harder to read; cuts against the "code as flow diagram" pitch |

---

### F. Wind speed

**Idea.** Each IP carries a `speed: BigInt` (initially `1`). On
every tick the IP advances `speed` cells in its current direction,
and **only the destination cell executes** — intermediate cells are
skipped (not decoded, no string-mode toggle, no unknown-glyph
warnings). Two new opcodes adjust speed: `≫` (GUST, U+226B,
`speed += 1`) and `≪` (CALM, U+226A, `speed −= 1`). Hitting
`speed == 0` from a CALM is a runtime error.

**Sketch.** Add `speed: BigInt` to `IpContext`. The §3.6 movement
rule becomes `position += dir * speed`. Two new opcodes in §4. `t`
SPLIT extended so the child inherits the parent's speed at split
time.

| | |
|-|-|
| Fit | ★★★★★ — *literal* wind metaphor: stronger wind, calmer wind |
| Novelty | ★★★★ — Befunge has nothing equivalent to variable-speed IPs |
| Spec cost | ★★ — one IP field, two opcodes, one rule change |
| Break risk | ★ — additive (default speed=1 reproduces v0.x exactly) |
| Payoff | ★★★★ — race patterns, skip-the-trap routing, deceleration puzzles |

This candidate was added during the v0.5 design review after A and
B were rejected (see *Recommendation* below).

## Recommendation

### Decided: **F (wind speed) + D (IP collision — merge)**

After review with the user, A and B were rejected:

- **A (inertia)** breaks too many v0.x programs. `_` and `|` flip the
  IP 180° routinely after a branch, and that routing pattern is
  everywhere. The compat flag `--free-winds` would have to live
  forever — meaning we'd be maintaining two languages forever.
- **B (time grid)** turns the grid into a 3D structure. The "you can
  see the whole program at once" pitch breaks. Visibility is a load-
  bearing identity claim for Windy and we can't trade it.

C (2D stack) reworks the data model wholesale; E (multi-token cells)
breaks the 1-cell-1-codepoint promise; D alone is too small to anchor
a major version cut.

A sixth candidate — **F (wind speed)** — was added during the review
and scores best on every axis (see *Scoring summary*): the most
thematically literal addition possible to a wind-named language,
purely additive (default speed=1 reproduces v0.x exactly), and a spec
cost of one IP field plus two opcodes.

**F + D together** fit naturally:

- F introduces speed differentials between IPs, which makes D's
  collision rule materially interesting — fast IPs can catch up to
  slow ones; races where the survivor is whoever produces the right
  merged direction.
- Both are additive: programs that never execute `≫` / `≪` and never
  produce a collision behave identically under v0.4 and v1.0.
- They are orthogonal in spec text: F adds one rule to §3.6 movement;
  D adds one collision pass after movement.

### Wind speed (F) — agreed details

- Each IP gains `speed: BigInt`, initially 1.
- Per-tick movement: `position += dir * speed`. **Only the destination
  cell executes** — intermediate cells are skipped (not decoded, no
  string-mode toggle, no unknown-glyph warnings).
- `≫` (U+226B, GUST): `speed += 1`.
- `≪` (U+226A, CALM): `speed -= 1`. Hitting 0 is a runtime error
  (a sharp edge — you must know your speed before you decelerate).
- BigInt rather than `u32`. Windy's identity is unbounded arithmetic;
  capping speed alone would be inconsistent. CALM is symmetric to
  GUST so deceleration is always available given enough operations.
- `t` SPLIT: child inherits parent's speed at split time.
- `~` TURBULENCE: speed preserved; only direction changes.

### IP collision (D) — agreed details, merge semantics

When two or more IPs occupy the same cell at end-of-tick (after
movement), they collapse into one:

- **stack**: concatenation in birth order, oldest at bottom.
- **direction**: vector sum, clipped to `{-1, 0, +1}` per axis. If
  the result is `(0, 0)` (head-on storm cancels itself), the merged
  IP **dies**.
- **speed**: max of constituents (strong wind absorbs weak).
- **strmode**: forced off. Merge is a fresh start.

End-of-tick coincidence only. Two IPs at speed 1 swapping positions
(adjacent, moving toward each other) pass through each other in v1.0;
mid-segment crossing detection is reserved for a future version.

## Open questions resolved

| Question | Resolution |
|----------|------------|
| Upper bound on speed | None — BigInt, consistent with Windy's unbounded-arithmetic identity. Symmetric CALM means deceleration is always reachable. |
| `≪` at speed 1 (would yield 0) | Runtime error. CALM needs a sharp edge; a no-op clamp would let programs decelerate lazily and erase the design tension. |
| Speed semantics — execute-everything vs skip-to-destination | Skip. Cleaner spec; "fast wind ignores obstacles" is on-theme; collision detection becomes well-defined as endpoint check. |
| `t` child speed | Inherits parent's speed at split time. |
| Merge stack order | Birth order; oldest IP's stack at the bottom. |
| Merge direction | Per-axis vector sum clipped to `{-1, 0, +1}`. `(0, 0)` ⇒ die. |
| Merge speed | Max of constituents. |
| Merge strmode | Forced off. |
| Mid-segment crossing detection | Out of scope for v1.0. Reserved for a future minor. |

## Outcome — what shipped

The plan above ran end-to-end and is now history; this memo's value is
the rationale that led to picking F + D. Concrete artefacts that
landed for the v1.0 cut:

- SPEC §3.7 (Wind Speed) and §3.8 (IP Collision — Merge) as
  normative sections; ≫ and ≪ in the §4 opcode table.
- `--v0` legacy gate on the CLI, the WASI binary, the `Session` /
  `run` wasm API (default `null` ⇒ v1), and the browser playground
  toolbar (checkbox default off ⇒ v1).
- `IpContext.speed: BigInt`, GUST / CALM dispatch, `pos += dir × speed`
  movement, post-tick collision pass — all gated on `Vm.v1`.
- `conformance/v1.json` (4 cases) loaded by `tests/conformance_v1.rs`
  with v1 mode ON, plus an additivity guard (`v0_cases_pass_under_v1_mode`)
  that re-runs every v0.4 conformance case under v1.0 to enforce the
  promise that programs without ≫/≪ and without collisions behave
  identically under both gates.
- `examples/gust.wnd` (speed mechanic), `examples/storm.wnd` (head-on
  merge → die).
- Browser debugger: per-IP speed badge in the state panel, exit 134
  surfaced as "trap" instead of a bare integer.

Asymmetric (right-angle) merge cases that exercise the stack-concat
rule are still on the wishlist — the harness has the head-on case but
not yet a layout that drives a non-fatal merge whose stack survives
into stdout. Reserved for v1.x examples.
