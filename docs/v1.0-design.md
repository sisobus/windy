# Windy v1.0 — design memo

> Status: **deciding**. v0.x has shipped. This document captures the
> options and an opinion; the actual cut is a follow-up.

## What v1.0 has to do

Through the v0.x line, Windy is — semantically — Befunge-98 with a
nicer surface and a few stricter promises. That's a fine *dialect*, but
"a new language" is a stronger claim than the spec currently earns.

For the major version to turn over, **Windy v1.0 must adopt at least
one semantic feature with no precedent in the Befunge family.** This
isn't about adding *more* — it's about adding *a thing the family
provably doesn't have*, so that "Windy is just Befunge with a haircut"
stops being a fair sentence.

Constraints any candidate has to clear:

1. **It has to fit the wind metaphor.** Windy is named after a wind
   mechanic; an addition that's tonally unrelated (e.g. an FFI shim) is
   the wrong move at this version cut.
2. **It can't paint over the existing 33 opcodes.** Programs written
   today should still run, with at most a documented compatibility
   note.
3. **It has to be tractably specifiable.** v0.x's whole identity is
   "the spec is the source of truth"; a v1.0 feature that requires a
   month of edge-case work before the spec is solid is a bad fit.
4. **It has to be *interesting to write programs against*.** The
   feature has to enable program patterns that didn't exist before, not
   just sit in §10 as a checkbox.

## Candidates

Five concrete directions. Each one is rated below on **fit** (does it
match the wind theme), **novelty** (does the Befunge family lack it),
**spec cost** (how much new SPEC text), **break risk** (do v0.x
programs still run), and **program-design payoff** (does it open new
patterns).

---

### A. Wind tension / inertia

**Idea.** The IP can no longer flip its direction freely. Direction
changes are constrained to the **graph of adjacent winds** — from any
of the eight winds you can rotate ±45°, but reversing 180° takes four
sequential turns. Existing direction-set opcodes (`→`, `↗`, `←`, etc.)
*request* a target direction; if that target is more than 45° from the
current heading the IP rotates one notch toward it and re-tries on the
next visit (or a new opcode `,` at a "drift" cell pre-rotates over a
gradient of cells).

**Sketch.** Add a single language-level invariant in §3:

> The IP's direction `(dx, dy)` is always one of the eight winds. A
> direction-set opcode that targets a wind more than one rotation away
> rotates the IP exactly one wind toward the target instead.

No new glyphs needed; the existing arrow set just gets re-interpreted.
Opt out via a `--free-winds` flag or a per-program directive for
backward compat.

| | |
|-|-|
| Fit | ★★★★★ — *defines* what "Windy" means |
| Novelty | ★★★★ — Befunge has nothing similar |
| Spec cost | ★★ — single rule, easy to write |
| Break risk | ★★★ — v0.x programs that flip 180° break unless `--free-winds` is on |
| Payoff | ★★★★ — routing becomes a real puzzle, programs look like wind currents |

---

### B. Time-aware grid

**Idea.** Cells are keyed `(x, y, t)`. The default `g` / `p` operate
at the current tick `t`. Two new opcodes, `G` and `P`, take an
explicit time offset:

```
y t G  →  push G[(x, y, t_now + t)]
v y t P →  G[(x, y, t_now + t)] := v
```

You can write a value into a *future* tick that no IP has reached yet,
or read a value that an IP wrote *in the past*. The grid stops being a
2D map and becomes a 3D recording.

**Sketch.** Two new opcodes, plus a §3.x on the time axis. The
already-sparse `HashMap<(x, y), BigInt>` becomes
`HashMap<(x, y, u64), BigInt>`; reads at `t_now` fall back to the
"all-time-default" plane (`HashMap<(x, y), BigInt>`) so initial
parsing of the source still produces values readable at every tick.

| | |
|-|-|
| Fit | ★★ — wind doesn't directly evoke time, but "weather forecasts" works |
| Novelty | ★★★★★ — esolangs with time-axis grids are rare |
| Spec cost | ★★★★ — needs a careful causal-consistency rule, paradox handling |
| Break risk | ★★ — purely additive opcodes |
| Payoff | ★★★★★ — opens *causal puzzles* foreign to Befunge |

---

### C. 2D stack

**Idea.** The stack is no longer a flat LIFO. It's a visible auxiliary
plane that grows in the direction of the IP at the moment of `push`:
heading east when you `push 5` puts the `5` one cell east of the
current "top" marker; heading north stacks it upward. `pop` reverses
the IP's last push direction.

**Sketch.** Replace §3.3 with a per-IP `stack_origin: (i64, i64)` and
`stack_top: (i64, i64)`. Push: `stack_top += dir; G[stack_top] = v`.
Pop: `v = G[stack_top]; G[stack_top] = SPACE; stack_top -= last_dir`.
The stack visually lives in a reserved region (e.g. `y < -1000`) and
the debugger renders it next to the program.

| | |
|-|-|
| Fit | ★★★ — wind currents pushing values around the plane is on-theme |
| Novelty | ★★★★ — Befunge has stack-of-stacks but not a *spatial* stack |
| Spec cost | ★★★★★ — every stack opcode (`+`, `:`, `\`, `$`, ...) needs new wording, multi-IP makes it hairy |
| Break risk | ★★★★★ — fundamentally changes data model |
| Payoff | ★★★★ — visual feedback in the debugger is gorgeous |

---

### D. IP collision semantics

**Idea.** When two live IPs land on the same cell on the same tick,
they don't pass through each other independently — they **interact**
per a defined rule. Likely options:

- **Annihilate**: both IPs die.
- **Merge**: stacks concatenate; direction becomes the (clipped) vector
  sum of the two; if the sum is zero, the merged IP dies.
- **Bounce**: each IP's direction reverses.

**Sketch.** Extends §3.5 with a "collision step" run after every
tick's IP pass. Probably needs a small tag on each IP so we can
distinguish "I started this tick at this cell" vs "I'm passing through
because two IPs collided here last tick".

| | |
|-|-|
| Fit | ★★★ — winds meeting can produce calm or storm; thematic |
| Novelty | ★★★★ — Befunge-98 has concurrent IPs but they ignore each other |
| Spec cost | ★★★ — well-bounded but needs careful tick semantics |
| Break risk | ★★ — only matters when `t` is used; v0.x split programs without collisions still work |
| Payoff | ★★★ — strategically interesting for multi-IP programs but not every program uses `t` |

---

### E. Cells as multi-token regions

**Idea.** A cell is no longer a single codepoint — it's a finite ordered
tuple of codepoints. The IP "consumes" one token per tick before
advancing. String mode (`"`) operates *inside* the cell tuple, not
across cells. Source files specify multi-token cells with a syntax like
`{abc}` to mean "a cell holding the tuple `('a', 'b', 'c')`".

**Sketch.** Big change to §3 (cell model) and §5 (file format). All
existing single-codepoint cells become 1-tuples implicitly; new syntax
opts into longer tuples.

| | |
|-|-|
| Fit | ★★ — doesn't really evoke wind |
| Novelty | ★★★ — bytecode-style packing in a 2D grid is unusual |
| Spec cost | ★★★★★ — touches parser, IR, debugger, file format |
| Break risk | ★★★ — syntax change; old files still parse |
| Payoff | ★★ — denser code, but harder to read; cuts against the "code as flow diagram" pitch |

## Recommendation

### Lead candidate: **A. Wind inertia**

The language is *named after* the wind mechanic. Adding a real
constraint to how the wind turns — instead of letting the IP flip 180°
on any cell — is the most thematically honest answer to "what makes
Windy not just Befunge?". It earns the name.

It's also the cheapest from a spec standpoint: a single rule in §3,
no new opcodes, no new syntax. Existing arrow glyphs just *mean
something* now.

The break risk is real (some v0.x programs flip the IP 180° in a single
cell — `_` and `|` do this routinely after a branch). The mitigation:
ship `--free-winds` as a v1.0 compat flag, default off, with a clear
deprecation horizon. Most v0.x programs only flip in routing
back-to-loop-entry, which is also the routing pattern that benefits the
most visually from gradient turns ("currents" that bend the IP rather
than slingshot it).

### Bold alternative: **B. Time-aware grid**

If we want the v1.0 cut to be a *headline* feature — esolang wiki entry,
HN post — time-axis cells are the most genuinely novel thing on the
list. Causal puzzles are barely explored in the esolang space, and the
implementation cost (one extra index in the hashmap) is laughably small
relative to the conceptual payoff.

The risk is that "Windy" stops being mainly about wind and starts being
mainly about time travel. That's not an obviously bad outcome — but it
*is* a different identity than the language has been pitching.

### Combination?

A and B are orthogonal. v1.0 could land **A** as the defining feature
("Windy is the wind-inertia esolang") and tease **B** for v1.x or v2.0
once the inertia rules have settled into program patterns we can build
on.

The combinations to *avoid* in a single cut:
- A + C — both touch direction semantics, hard to keep coherent.
- D alone — interesting but not enough to anchor a major version.
- E with anything — multi-token cells fight the visual flow we're
  doubling down on with A.

## Open questions before we cut

1. **Is `--free-winds` permanent, or does it sunset?** If permanent we
   maintain two languages forever; if it sunsets we need a deprecation
   schedule.
2. **What does inertia do to `~` (TURBULENCE)?** Does the random pick
   ignore inertia (pure chaos), or does it pick from the two adjacent
   winds (constrained chaos)? The latter fits better.
3. **Should `_` and `|` be re-spec'd to use one-step rotation, or stay
   "set absolute"?** Probably the former for consistency, but it
   deserves an explicit decision.
4. **How does inertia interact with `t` (SPLIT)?** The new IP is born
   going `(-dx, -dy)` — that's a 180° flip from the parent. Is that
   still legal under inertia? Probably yes (spawn-time direction is a
   design choice, not a rotation), but spec-state it.

## Next concrete steps

1. Write a draft §3.x for wind inertia in `SPEC.md`, behind a
   `## Pre-release: v1.0 (proposal)` header that doesn't affect v0.x
   conformance.
2. Add a tiny `examples/inertia.wnd` that's only correct under the
   inertia rule — a regression target.
3. Implement the inertia rule behind a `Vm::with_inertia` builder
   method; gate via `--inertia` CLI flag in addition to `--free-winds`.
4. Run conformance: every v0.x test must still pass with default
   (free-winds) flags. Inertia tests live in a parallel
   `conformance/inertia.json` that only the new mode loads.
5. After two releases of stable inertia, bump the spec to v1.0 and
   make inertia the default; `--free-winds` sticks around as the
   compat escape hatch.
